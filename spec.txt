Gallopics API
Functional Requirement
Architecture & Specification
Project: Gallopics API
Phase: Release 1
Version: 1.0

1) Recommended architecture
Use Python + FastAPI.
Core stack
fastapi → API layer
uvicorn → ASGI server
httpx → async integrations (TDB, Equipe, Klarna)
pydantic → schemas
sqlalchemy + alembic → ORM + migrations
psycopg[binary] → PostgreSQL driver
redis → caching + rate limiting
tenacity → retries
python-slugify → URL-safe slugs
rapidfuzz → matching logic
python-dateutil → date parsing
structlog → structured logging
Storage
PostgreSQL → primary DB
Object storage (S3 / R2 / Azure Blob) → images
Auth
Clerk
backend JWT/session validation
no custom auth system
Background processing
Celery or RQ (recommended)
jobs:
TDB sync
Equipe sync
event matching
image processing (thumbnails, watermark)

2) Overview
Backend is a normalization + orchestration layer, not a proxy.
It must:
ingest events from TDB
ingest results from Equipe
match and merge events
store normalized data
expose stable API for gallopics.com
handle users (Clerk)
handle payments (Klarna)
handle photographer uploads + galleries
hide all external API complexity

3) Functional requirements
3.1 Event aggregation
Fetch events from TDB (search endpoint)
Normalize into internal schema
Store raw payload for traceability
Support filters:
date range
discipline
horse type
district
organizer
status
sustainability
Cache responses (Redis)
Scheduled sync (hourly/daily)

3.2 Event ↔ Result matching
Fetch meetings from Equipe API
Match using priority:
tdb_id
date + normalized name
organizer similarity
venue similarity
fuzzy score
Store:
match_status
match_score
match_method
Expose merged event + results

3.3 Website consumption
Frontend must not call external APIs.
It consumes:
events
results
galleries
checkout
orders
user profile

3.4 User identity (Clerk)
sign up / login handled by Clerk
backend validates JWT
store local user:
clerk_user_id
role (user, photographer, admin)

3.5 Payments (Klarna)
Support full lifecycle:
create payment session
authorize payment
create internal order
capture
refund
cancel
Store:
klarna_session_id
authorization_token
order_id
Critical:
idempotency required
server-side callback required

3.6 Photographer module (NEW – REQUIRED)
Upload
batch upload via presigned URLs
no direct file upload via API
Metadata
tag photos:
rider
horse
class
start number
Visibility
draft / published / hidden
Processing
generate:
thumbnail
preview
watermark
Sales
price per photo
Klarna checkout for photos

3.7 Admin
manual event matching
re-run sync/matching
manage orders
refund/cancel payments
moderate photos
manage photographers

3.8 Non-functional
rate limiting
retry + timeout (external APIs)
audit logging
idempotency (orders)
API versioning /api/v1
background jobs
observability (logs + metrics)

6) Matching logic (final)
Priority:
tdb_id → exact match
date match
normalized name
organizer similarity
venue similarity
fuzzy match
Scoring
1.00 → exact id
0.85 → date + strong name
0.75 → date + partial match
<0.70 → reject
Store
match_status
match_score
match_method
